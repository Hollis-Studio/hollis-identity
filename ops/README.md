# Operations — Hollis Identity Service

This runbook covers infrastructure provisioning, deployment, database migrations, secret management, and routine operational tasks for `hollis-identity`.

---

## Infrastructure overview

All infrastructure is managed by Terraform in `infrastructure/`. There is no hand-maintained `ops/ecs-task-def.json` — ECS task definitions are generated by Terraform.

Resources provisioned:

| Resource | Details |
|---|---|
| ECR repository | Container image registry; scan on push enabled |
| VPC | `10.42.0.0/16`; 2 public subnets (ALB) + 2 private subnets (ECS + RDS) |
| ALB | Application Load Balancer; HTTP listener always created; HTTPS listener created when `certificate_arn` is set |
| ECS Fargate | Cluster, task definition, service; default 2 tasks, 512 CPU / 1024 MiB |
| RDS PostgreSQL 16 | `db.t4g.micro` by default; encrypted storage; Multi-AZ flag per environment |
| Secrets Manager | Two secrets: `{name}/app/` (JWT keys, ENCRYPTION_KEY, PASSWORD_PEPPER) and `{name}/database/` (DATABASE_URL) |
| RS256 JWT key pair | Generated by Terraform (`tls_private_key`); stored in Secrets Manager |
| CloudWatch Logs | `/ecs/{name}`; 30-day retention (dev), 90-day (prod) |
| WAF | Rate-based rule on ALB; default 1000 req/5 min per IP |

## Terraform — plan and apply

```bash
cd infrastructure

# First time setup
cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars with environment-specific values

terraform init
terraform plan
```

**Before applying production**, set the following in `terraform.tfvars`:

```hcl
environment          = "prod"
db_multi_az          = true
certificate_arn      = "<ACM ARN in us-east-1>"
identity_domain_name = "identity.hollis.health"
reset_password_url   = "https://hollis.health/reset-password"
```

Also configure a production-specific Terraform state backend before applying.

```bash
terraform apply
```

After apply, Terraform outputs the ECR repository URL and ALB DNS name.

## Container — build and push

The Dockerfile is a three-stage build. It requires a BuildKit npmrc secret for GitHub Packages access.

```bash
# Build
DOCKER_BUILDKIT=1 docker build \
  --secret id=npmrc,src=.env.npm.local \
  -t <ecr_repository_url>:<image_tag> .

# Authenticate with ECR
# <!-- UNVERIFIED: replace <account-id> and <region> with actual values after apply -->
aws ecr get-login-password --region us-east-1 \
  | docker login --username AWS \
    --password-stdin <account-id>.dkr.ecr.us-east-1.amazonaws.com

# Push
docker push <ecr_repository_url>:<image_tag>
```

Set `image_tag` in `terraform.tfvars` to the pushed tag, then re-run `terraform apply` to update the ECS task definition.

## Secret management

Terraform creates two Secrets Manager secrets and populates them on first apply:

| Secret path | Contents |
|---|---|
| `hollis-identity-{env}/app/...` | `JWT_SECRET`, `JWT_PRIVATE_KEY`, `JWT_PUBLIC_KEY`, `JWT_KEY_ID`, `ENCRYPTION_KEY`, `PASSWORD_PEPPER` |
| `hollis-identity-{env}/database/...` | `DATABASE_URL` (with `sslmode=require&connection_limit=20&pool_timeout=10`), host, port, username, password |

The ECS task definition pulls individual fields from these secrets via `secrets[].valueFrom` ARN references. No environment variables are hard-coded with secret values.

### Rotating secrets manually

```bash
# <!-- UNVERIFIED: actual secret ARNs are assigned by Terraform apply -->
aws secretsmanager put-secret-value \
  --secret-id <secret-arn> \
  --secret-string '{"JWT_SECRET":"<new-value>", ...}'
```

After rotating JWT keys, existing access tokens (15-minute TTL) and refresh tokens (7-day TTL) signed with the old key will fail verification. Plan for a rolling token invalidation window or force-logout all users.

## Health checks

The ALB health check polls `GET /health` on port 4001 every 30 seconds. The endpoint performs a live database ping (`SELECT 1`):

- `200 { ok: true, service: "hollis-identity", db: "ok" }` — healthy
- `503 { ok: false, service: "hollis-identity", db: "unreachable" }` — DB connection failed; ALB will drain and replace the task

Thresholds: 2 consecutive successes → healthy; 3 consecutive failures → unhealthy.

## Log access

CloudWatch log group: `/ecs/hollis-identity-<env>`

```bash
# Tail logs (requires awslogs or aws-cli)
aws logs tail /ecs/hollis-identity-<env> --follow

# Filter for security events
aws logs filter-log-events \
  --log-group-name /ecs/hollis-identity-<env> \
  --filter-pattern '"[SECURITY]"'

# Filter for MFA events
aws logs filter-log-events \
  --log-group-name /ecs/hollis-identity-<env> \
  --filter-pattern '"[MFA]"'
```

All logs are structured JSON (Pino). Each request log includes `requestId` which also appears in the `X-Request-Id` response header and the JSON error body — use it to correlate client-reported errors with server logs.

## Token revocation runbook

### Revoke a single user's sessions (e.g., account compromise)

The `denyAllUserAccessTokens` function writes a watermark to `UserTokenDenylistEntry`. All access tokens issued at or before the watermark are rejected. Refresh tokens must be revoked separately in `RefreshToken`.

This can be triggered programmatically or via a one-off script against the database:

```sql
-- Revoke all refresh tokens for a user
UPDATE "RefreshToken"
SET "revokedAt" = NOW(), "revokedReason" = 'admin_action'
WHERE "userId" = '<user-id>'
  AND "revokedAt" IS NULL;

-- Insert a user-level access token denylist entry
-- (expiresAt = now + 15 minutes covers all outstanding access tokens)
INSERT INTO "UserTokenDenylistEntry" ("id", "userId", "deniedBefore", "reason", "expiresAt")
VALUES (gen_random_uuid(), '<user-id>', NOW(), 'admin_action', NOW() + INTERVAL '15 minutes')
ON CONFLICT ("userId") DO UPDATE
  SET "deniedBefore" = NOW(),
      "reason" = 'admin_action',
      "expiresAt" = NOW() + INTERVAL '15 minutes',
      "revokedAt" = NOW();
```

### Clear expired denylist entries manually

```sql
DELETE FROM "AccessTokenDenylistEntry" WHERE "expiresAt" < NOW();
DELETE FROM "UserTokenDenylistEntry"   WHERE "expiresAt" < NOW();
```

The service also runs automatic cleanup on a 1-minute interval via `store.startCleanupTimer()`.

## WAF

AWS WAF is attached to the ALB with a rate-based rule (default 1000 requests per 5 minutes per IP). The limit is configurable via `waf_rate_limit` in `terraform.tfvars`.

The application-level rate limiters (`loginRateLimiter`, `authSessionRateLimiter`) are a defense-in-depth measure — WAF is the primary protection against distributed abuse.

## SES email setup

Email delivery is wired through AWS SES when `EMAIL_PROVIDER=ses`.

Before password reset or verification emails will deliver:

1. Verify the sender domain or address in SES (<!-- UNVERIFIED: domain/address to verify — depends on production `email_from` value -->).
2. If the account is in SES sandbox, request production access.
3. Confirm `AWS_REGION` matches the region where the SES identity is verified.
4. Confirm `RESET_PASSWORD_URL` points at the frontend reset-password page (not the Identity API).

The ECS task IAM role receives `ses:SendEmail` and `ses:SendRawEmail` permissions via Terraform when `email_provider = "ses"`.

## DNS and TLS

<!-- UNVERIFIED: Route53 hosted zone and ACM certificate — must be created and ARN supplied before Terraform can create the HTTPS listener -->

1. Create an ACM certificate in `us-east-1` for `identity.hollis.health` (or the production domain).
2. Set `certificate_arn` in `terraform.tfvars` and re-apply.
3. Create a Route53 CNAME or alias record pointing the identity domain at the ALB DNS name from Terraform output.

The HTTPS listener uses TLS policy `ELBSecurityPolicy-TLS13-1-2-2021-06` (TLS 1.2+ with TLS 1.3 preferred).
